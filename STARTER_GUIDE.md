## このガイドの位置づけ

Zee Starter Guide は、AI 時代の防御準備に不安を感じるすべての組織と個人が、**最初の一歩を踏み出す** ためのものです。

ここで扱うのは、ツール選定や具体的な製品の比較ではありません。**自組織の現在地を知り、何を考えるべきかを整理する** ことに集中します。実装や運用の前に、まず「何を守りたいのか」を言葉にすることが、Zee が考える最初の一歩です。

このガイドは特定の業種・規模を対象にしません。AI 時代の攻撃に不安を感じるすべての組織と個人を想定しています。

---

## 6 つの起点

### 1. 不安を整理する

「何が不安か」を、技術用語を使わずに 1 ページで書き出してください。
顧客情報の漏えい、過去の取引履歴の改ざん、業務停止、評判の毀損、賠償、規制違反 —— それぞれの組織で具体的なものが違います。

不安の中身がはっきりしないまま対策を選ぶと、対策が効果的かどうかを後から判定できません。

### 2. 「絶対に盗まれたくない情報」をリスト化する

完全な棚卸しは不要です。**もしこれが流出したら、業務の根幹が崩れる** という情報だけを 3〜10 件挙げてください。

- 顧客の個人情報・取引履歴
- 設計図・ソースコード・処方・レシピ
- 認証情報・API キー・社内アクセス権
- 経営情報・未公開戦略・M&A 関連

このリストが、後で「Zee の罠でどれを守るか」を決めるときの根拠になります。

### 3. 入口防御の現状を知る

現時点で導入されている入口防御（ファイアウォール・EDR・パッチ運用・MFA など）を箇条書きで列挙してください。「導入しているつもり」と「実際に動いている」は別物です。

最終更新日・有効状態・カバー範囲を簡単に書き出すだけで、抜けている層が見えます。

### 4. 検知の存在を確認する

「もし侵入されたら、誰がいつ気づくか？」を 1 行で書いてください。

書けない場合、それが現状です。Zee は侵入を防ぐツールではなく、**侵入された後** に **狭く高信頼な検知信号（囮への接触）** を 1 本足す層です。「あなたの検知体制 **全般**」を肩代わりするものではありませんが、組織の検知能力に **特定の高信頼な信号** を新たに加えます。受け取った信号を人が見る運用がゼロだと、Zee の役割が成立しません。検知ゼロから少しでも上に進むのが、最初の意味のあるステップです。

### 5. 復旧の段取りを書く

侵入が確認された日の手順を、3 行で書いてください。

- 誰が初動を取るか
- 誰に連絡するか
- 何を止め、何を残すか

書けない場合、それが最大のリスクです。技術的な対策より先に、人と連絡の手順を整える価値があります。

### 6. Zee の関与する範囲を理解する

最後に、Zee がこの 5 つの中でどこに位置するかを確認してください。

- Zee は **③⑤** には触りません（入口防御、復旧の人手の部分）
- **④（検知）** について、Zee は「あなたの検知体制 **全般**」を肩代わりしません。代わりに **狭く高信頼な検知信号（囮への接触）** を 1 本足します。これが Zee の一次価値です
- Zee が向き合うのは、**侵入された後** の層 —— 盗まれにくくする、時間を稼ぐ、狭く高信頼な信号を出す
- Zee が他の層を置き換えることはありません

## 7. dry_run を試してみる

MVP は既定で **dry_run** で動きます。実遮断はせず、「もし遮断するならいつだったか」を計測ログに残すだけです。これは「囮の精度・誤検知率を、害なく測る」期間です。

最初の試行はこの順序を強く推奨します：

1. `examples/assets.example.toml` を `./assets.toml` にコピーする
2. `decoy_paths` を Zee 専用ディレクトリ配下のダミーパスに書き換える（例：`~/Documents/zee-decoys/aws-credentials.decoy`）。`~/.aws/` などの正規ツールのディレクトリへの配置は誤検知温床になるため避ける
3. `response_mode: notify` のまま起動（最初は遮断系の挙動も dry_run も走らせない）
4. 数日〜1 週間、自分の業務をいつも通りやってみる。誤検知（自分のバックアップツールや IDE が囮を踏むなど）が **ゼロ** であることを確認する
5. 誤検知ゼロが確認できたら、資産ごとに `response_mode: staged` か `auto` に昇格する

「すぐ auto にする」は **おすすめしません**。Zee は自分の業務を巻き込んだ瞬間に信頼を失う層です。誤検知ゼロを観測してからの昇格が、Zee を長く使うための前提です。

### auto に昇格する前に — `zee init-restore-token`

`response_mode: auto` で実遮断が走ったあと、ネットワークを復旧させる唯一の手段が `zee restore` です。v0.3 から、このコマンドは **復旧トークン** を要求するようになりました（CLI を実行できる別シェルからの誤った巻き戻しを塞ぐため）。

`auto` 昇格の **前に一度だけ** 次を実行し、表示されたトークンを安全な場所（パスワードマネージャ等）に控えてください：

```bash
zee init-restore-token
```

復旧時は `zee restore <asset_id> --token <TOKEN>` または `ZEE_RESTORE_TOKEN=<TOKEN> zee restore <asset_id>` で実行します。トークンファイル `~/.zee/restore_token` は 0600 で書かれます。マルチユーザー本番では `sudo` で `zee restore` を囲うか、専用ユーザーで運用してください（root 相当の攻撃者はトークンファイルを読めるため）。

### 自動遮断の発火条件（v4・必読）

`response_mode: auto` ＋ `dry_run: false` に昇格しても、自動遮断が走るのは **囮への「変更系」操作（書き換え・削除・改名・拡張）だけ** です。**囮への「読み取り系」操作（open / read / 属性参照）は通知のみで、自動遮断しません。**

理由は単純です。正規のバックアップ・AV・ファイルインデクサ等の一括リーダーは、囮を **読む** ことはあっても **書き換える** ことはしません。プロセス特定ができない現バックエンド（Linux inotify / macOS kqueue / Windows ReadDirectoryChangesW）で、操作の種類だけを根拠に「正規ソフトが普通やらない動き」を区別するのが、構造的に最も安全な誤検知制御です。

読み取り通知が来たら、心当たりがない場合は手動で `zee cut <asset_id>` を実行してください（復旧は `zee restore <asset_id>`）。「正規ソフトなので無視してよい」と断定するヒントは Zee 側からは出しません。最終判断は人間に委ねます。

macOS / Windows の「読むだけ」の攻撃を観測するには、`ZEE_CANARY_BASE_URL` 環境変数に外部エンドポイント（自前 webhook 受信器・[Canarytokens.org](https://canarytokens.org)・AWS Lambda 等）の URL を設定して `zee watch` を起動します。設定すると seeder が囮ファイルに canary URL を自動埋め込みし、攻撃者がその URL を辿った瞬間に operator 側で発火します（Zee 本体には戻りません）。未設定では canary URL は埋め込まれず、macOS / Windows での「読むだけ」の攻撃は観測されません（既定で安全側）。Linux は inotify が read を直接観察するので canary 不要です。
