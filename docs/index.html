不安から、備えへ。
高度AI時代の、最初の一歩。

Zee は、高度AI時代に防御準備が追いついていない組織と個人が、最初の一歩を踏み出すためのオープンプロジェクトです。侵入を防ぐ道具ではなく、盗まれにくくし、時間を稼ぐことを設計目標にしています。

なぜ Zee があるのか

社会の不安を、少しでも減らしたい。

AIは素晴らしい技術です。しかし攻撃側もそれを使えます。問題はAIそのものではなく、社会の準備が追いついていないこと。Zeeが向き合うのは、攻撃能力の進化速度と、防御準備の普及速度の差です。

Zeeが有名になることが成功ではありません。社会が安定すれば成功です。誰かがより良い方法を作るなら、それでも良いと考えています。

Mythos 時代の備え

機械の速度で動く攻撃が現れた。

脆弱性の「発見」と「武器化」が、安く・速く・大量になっています。以下はいずれも、AIを開発するAnthropic自身が公表した数字です。

MVP — 動く囮トリップワイヤ

既定 dry_run。明示昇格まで実遮断はしない。

Zee は侵入を防ぐ層ではなく、侵入後に狭く高信頼な検知信号（囮への接触）を1本足す層です。「あなたの検知体制全般」を肩代わりするものではありません。

# 1. インストール git clone https://github.com/KAMANOI/zee.git cd zee pip install -e . # 2. 資産プロファイルを作成 cp examples/assets.example.toml ./assets.toml # 編集して decoy_paths を自組織のパスに置き換える # 3. 監視開始（既定は dry_run、実遮断なし） zee watch -c ./assets.toml # 4.（任意）macOS / Windows での「読み取り検知」を有効化 # 外部 webhook 受信器（Canarytokens.org / 自前 / Lambda 等）の URL を渡す export ZEE_CANARY_BASE_URL="https://your-receiver.example.com/r" # 5. 復旧トークンを 1 回だけ生成（v0.3 以降必須） zee init-restore-token # 表示されたトークンを安全な場所に控える # 6. 別ウィンドウで「攻撃者の改ざんを真似て」囮を書き換える echo "tampered $(date)" >> ~/Documents/zee-decoys/.env # → 変更系の接触として 3 OS 共通で高信頼イベントを記録します。 # Linux では `cat <decoy>` の読み取りだけでも記録します。 # macOS / Windows の読み取りは ZEE_CANARY_BASE_URL 設定時のみ # 観測されます（埋め込んだ canary URL が外部受信器で発火）。 # 7.（遮断が走った後）復旧 zee restore <asset_id> --token <TOKEN> # または環境変数経由：ZEE_RESTORE_TOKEN=<TOKEN> zee restore <asset_id>

macOS・Windows は OS の制約で「読み取り検知」が直接できません。環境変数 ZEE_CANARY_BASE_URL に外部 webhook 受信器の URL を設定すると、seeder が囮ファイルに canary URL を埋め込み、攻撃者がその URL を辿った瞬間に外部受信器で発火します（Zee 本体には戻りません・out-of-band）。未設定では canary URL は埋め込まれず、macOS / Windows での「読むだけ」の攻撃は観測されません（既定で安全側）。Linux は inotify が読み取りも直接観察するので canary 不要です。
受信器の例：Canarytokens.org（無料・推奨）／自前 Cloudflare Worker・AWS Lambda・Vercel Edge Function 等。

カバレッジ

何を守り、何を守らないか。

※「防御率○%」のような数字は出しません。実環境はそれほど単純でなく、誇張は信頼を損なうためです。

対応する
  • 侵入後の異常検知（囮への接触＝高信頼イベント）
  • OS別の自動封じ込め（full / egress・既定 dry_run）
  • レイテンシ・op_class（read / change）・誤検知マーカーの計測（decoy_ref で位置匿名化）
  • 誤作動の最小化（配置ガイド ＋ 変更系操作のみ自動遮断）
  • 能力マトリクスでの正直な OS 別開示
対応しない（正直に）
  • 侵入そのものの阻止入口防御（ファイアウォール・EDR・パッチ）の役割。Zeeは置き換えではなく補完
  • あなたの検知体制全般の肩代わりZeeは「狭く高信頼な信号」を1本足す層
  • ネットワーク層での攻め返し（C2汚染）現時点では未実装
  • 脆弱性の修正・パッチ適用Zeeは検知と無力化の層。修正は別途必要
誤作動を防ぐ設計

正規の社員を“攻撃者扱い”しないことを最優先に。

侵入後で作動する層の最大のリスクは、正規のプロセスや社員を誤って罠に引き込んでしまうことです。Zeeはこのリスクを最小化することを目指します（「ゼロ」は約束しません）。

  • 検知と引き込みの間に二重の安全装置を置く。
  • 「誤作動の最小化」と「重大な攻撃の検知」という相反する要件を、1つの仕組みに詰め込まず役割を分けて両立。
  • 判定は AI の裁量ではなく確定的なルールで行う。
  • 初期は全資産 dry_run。誤検知ゼロを観測してから資産ごとに昇格する運用を Starter Guide で案内。
プライバシー

取得しない設計。

  • 現 MVP は 攻撃者の IP を取得・保存しません（watcher / responder / telemetry のいずれの経路でも IP やPII を扱いません）。記録するのは「どの囮に・いつ・どの種類（read / change）の接触があったか」と遮断レイテンシだけです。
  • canary URL は operator が指定した外部エンドポイント（自前 webhook 受信器・Canarytokens.org・AWS Lambda 等）で発火します。発信元 IP がそこで捕捉されえますが、それは Zee の外側であり、operator が選んだ外部エンドポイントのプライバシー方針の責任範囲です。
  • 蓄積データは期限を過ぎると自動的に削除する（設計目標）。
  • 相関を追加する場合は「個人」ではなく「セッション」を軸とする。

将来 IP 取得経路を追加する場合の方針（末尾オクテット切り捨てを第一候補とし、ハッシュ化は逆引き耐性が崩れるため採用しない）は ARCHITECTURE.md を参照してください。

Limitations — Zee がやらないこと

失敗条件を先に書いておきます。

  • 侵入そのものは防がない — 入口防御（ファイアウォール・EDR・パッチ）の役割を置き換えるものではありません。
  • 検知中心・既定 dry_run — 自動の封じ込め機構は資産プロファイルで response_mode: auto を明示し dry_run: false にしたときだけ動きます。
  • 自動遮断は「変更系の接触」のみ — open / read / 属性参照などの読み取り系は自動遮断しません。通知のみで、必要なら zee cut <asset_id> で手動遮断します。プロセス特定ができない現バックエンドでは、正規ソフトが普通やらない操作だけを自動遮断の根拠にするのが構造的に安全だからです。
  • macOS / Windows の読み取り検知は ZEE_CANARY_BASE_URL 設定時のみ — kqueue / ReadDirectoryChangesW は read を通知しません。環境変数を設定すると seeder が canary URL を decoy に自動埋め込みし、外部受信器で発火します（Zee 本体には戻りません）。未設定では canary URL は埋め込まれず、macOS / Windows での「読むだけ」の攻撃は観測されません。Linux は inotify で read を直接観察するので canary 不要です。
  • プロセス allowlist は現バックエンドでは効かない — 誤検知制御は「配置ガイド」＋「変更系トリガー限定」で行います。allowlist 路線は v4 で撤回しました（v2 で同梱した既定 allowlist と auto+dry_run:false ハードブロックも撤回）。
  • zee restore には restore_token が必要（v0.3 から） — zee init-restore-token で生成したトークンを --token または ZEE_RESTORE_TOKEN で渡します。同一ユーザーで動作する別シェルからの誤った巻き戻しは塞げますが、~/.zee/restore_token（0600）を読める root 相当の攻撃者にはバイパスされます。マルチユーザー本番では sudo で囲うか専用ユーザーで運用してください。
  • イベントログは decoy_ref（asset_id#index）で記録（v0.3 から） — 絶対パスは記録されないため、ログを読まれても囮位置は一括列挙できません。対応関係は assets.toml 経由でのみ復元できます。
  • 限界がある — 機械の速度で動く相手や、ごく小さく具体的な秘密（単一の API キーなど）の保全は、Zee の範囲を超えます。
  • 未測定 — 効果はまだ独立に検証されていません。本番運用の前に、ご自身の環境で必ず確認してください。

これは床であって天井ではありません。安全だと言う前に、失敗条件を先に書きました。

みんなで防災 — Defense by everyone, for everyone

Zee は seed 型 OSS。育てるのは皆。

Zee はメンテナがリファレンス実装と設計を公開して終わりにする seed（種まき）型 OSS です。あなたの環境で動かす、OS の更新に追従させる、業種に合わせて改造するのは 各自。一人の管理者が永続的に全 OS を追従する OSS のメンテナ疲れ問題から最初から外れます。

代わりに、修正の知恵を皆で共有する場 が用意されています。次に同じ問題に当たった人の作業を 3 時間 → 30 秒 に縮めるのが目的です。
